# NovelTrans UI/UX 전면 개편안

## 1. UI/UX 방향 전환

기존 구조는 wizard 기반이다. 사용자가 순서대로 질문에 답하고, 선택지를 고르고, 다음 단계로 넘어가는 방식이다. 이 방식은 구현이 쉽고 안정적이지만, 제품 인상은 딱딱하다. 장편 번역 도구가 아니라 설정 많은 콘솔처럼 느껴진다.

NovelTrans는 wizard를 버리고 **Translation Studio UX**를 채택한다.

핵심 컨셉은 다음과 같다.

> 사용자는 번역을 “실행”하는 것이 아니라, 하나의 작품을 “작업실에 올려놓고 관리”한다.

따라서 화면은 메뉴 트리가 아니라 작업실 구조를 가진다.

```text
Bookshelf → Studio → Glossary Lab → Review Desk → Export Room
```

* Bookshelf: 프로젝트 보관함
* Studio: 번역 작업실
* Glossary Lab: 용어집 연구실
* Review Desk: 검수 작업대
* Export Room: 결과물 제작실

## 2. 제품 인상

NovelTrans의 UI는 다음 인상을 목표로 한다.

* 차갑고 관리도구 같은 CLI가 아니다.
* 게임처럼 과하게 꾸민 UI도 아니다.
* 장편 번역을 다루는 전문 작업실 느낌이다.
* 상태, 진행률, 용어집, 문제점이 한눈에 보인다.
* 사용자는 “다음에 뭘 해야 하지?”를 고민하지 않는다.
* 앱이 항상 다음 행동을 제안한다.

키워드:

```text
작업실
책장
실험실
검수 큐
번역 레시피
용어 맥박
프로젝트 타임라인
```

## 3. 전체 정보 구조

기존 메뉴형 구조:

```text
새 번역
이어서 번역
결과 파일 다시 만들기
도구와 설정
```

새 구조:

```text
Bookshelf
  ├─ 최근 프로젝트
  ├─ 새 작품 가져오기
  ├─ 멈춘 작업 이어가기
  └─ 문제 있는 프로젝트

Studio
  ├─ 원문 상태
  ├─ 번역 진행
  ├─ 현재 큐
  ├─ 실시간 이벤트
  └─ 다음 추천 행동

Glossary Lab
  ├─ 후보 용어
  ├─ 충돌 용어
  ├─ 잠긴 용어
  ├─ 금칙 번역어
  └─ 용어집 품질 점수

Review Desk
  ├─ QA 이슈
  ├─ 누락 의심
  ├─ 일본어 잔존
  ├─ 이름 흔들림
  └─ 수정 후 재검사

Export Room
  ├─ TXT
  ├─ EPUB
  ├─ 용어집 부록
  ├─ 세로쓰기
  └─ 결과 미리보기
```

## 4. 첫 화면: Bookshelf

앱을 실행하면 메뉴가 아니라 프로젝트 책장이 열린다.

```text
NovelTrans
Translation Studio

┌─ Continue ───────────────────────────────────────────────┐
│  마도서관의 사서                                          │
│  72 / 100화 완료   용어 충돌 5개   마지막 작업 2시간 전     │
│  [Enter] 이어가기   [G] 용어집   [R] 검수                   │
└──────────────────────────────────────────────────────────┘

┌─ Recent Projects ────────────────────────────────────────┐
│  1. 북쪽 변경의 기사     완료 18%   후보 용어 21개          │
│  2. 용사의 귀환          완료 100%  EPUB 생성됨             │
│  3. 검은 숲의 마녀       실패 3화   재시도 필요             │
└──────────────────────────────────────────────────────────┘

[N] 새 작품 가져오기   [/] 검색   [S] 설정   [?] 도움말
```

첫 화면의 목적은 단순하다.

* 새 작업 시작
* 기존 작업 이어가기
* 문제가 있는 작업 복구
* 최근 프로젝트 빠르게 접근

사용자는 “새 번역인지 이어 번역인지” 먼저 고를 필요가 없다. 앱이 최근 상태를 보여주고 다음 행동을 제안한다.

## 5. 새 작품 가져오기 UX

새 번역은 긴 wizard가 아니라 **Drop-in Flow**로 간다.

```text
새 작품 가져오기

원문 파일을 끌어오거나 경로를 붙여넣으세요.
또는 본문을 바로 붙여넣어도 됩니다.

› ~/Downloads/source.txt
```

입력 후 앱은 바로 분석한다.

```text
원문 분석 완료

제목 추정: 마도서관의 사서
화수: 100화
총 글자 수: 1,284,220자
언어: 일본어
구조: 화 제목 감지됨
위험 요소: 작가 후기 18개, 긴 화 4개

추천 번역 레시피:
균형 번역 · 4화 병렬 · 용어집 엄격도 높음 · TXT+EPUB

[Enter] 이대로 시작
[E] 레시피 수정
[G] 먼저 용어집 훑기
```

핵심은 설정을 먼저 묻지 않는 것이다.
앱이 먼저 분석하고, 추천값을 제안하고, 사용자는 수정할 때만 들어간다.

## 6. 번역 레시피

“설정”이라는 단어 대신 **번역 레시피**를 사용한다.

번역 레시피는 한 프로젝트의 번역 성향과 실행 방식을 요약한 카드다.

```text
번역 레시피

문체        한국 웹소설 균형체
용어집      확정 용어 강제, 후보 자동 수집
속도        4화 병렬
검수        누락 / 일본어 잔존 / 용어 흔들림
출력        TXT, EPUB
백엔드      OpenAI-compatible
```

프리셋은 다음처럼 제공한다.

```text
1. 빠른 초벌 번역
2. 균형 번역
3. 문학적 자연화
4. 직역 보존
5. 용어 일관성 우선
6. 커스텀
```

NovelTrans의 핵심 특화가 용어집이므로, 기본 추천은 “균형 번역 + 용어집 엄격도 높음”으로 둔다.

## 7. Studio: 메인 작업실

번역 실행 화면은 단순 로그 창이 아니다.
작품 하나를 중심에 둔 작업실이어야 한다.

```text
마도서관의 사서
Translation Studio

┌─ Progress ───────────────────────┬─ Live Events ─────────────────────┐
│  ████████░░░░░░░░  48%            │  48화 번역 완료                   │
│                                   │  49화 QA 검사 중                  │
│  완료      48                     │  용어 후보 3개 발견               │
│  진행 중   49, 50, 51, 52          │  “黒架” 번역 충돌 감지             │
│  대기      53-100                 │  EPUB 목차 초안 업데이트           │
│  실패      없음                   │                                   │
├───────────────────────────────────┴───────────────────────────────────┤
│  Next Action: 용어 충돌 1개가 있습니다. [G] 지금 검토                  │
└───────────────────────────────────────────────────────────────────────┘

[Space] 일시정지   [G] 용어집   [R] 검수   [E] 내보내기   [Q] 종료
```

중요한 점은 로그를 단순히 흘려보내지 않는 것이다.

앱은 이벤트를 해석해서 “다음 행동”을 제안한다.

예시:

```text
Next Action:
- 용어 충돌 1개가 있습니다. 지금 확정하면 이후 화에 반영됩니다.
- 3개 화에서 일본어 잔존이 감지되었습니다. 번역 후 Review Desk에서 확인하세요.
- 현재 속도 기준 약 18분 남았습니다.
```

## 8. 용어 맥박: Glossary Pulse

번역 중에도 용어집 상태가 살아 움직여야 한다.

Studio 화면에는 항상 용어집 요약을 표시한다.

```text
Glossary Pulse

확정 용어      128
잠긴 용어       34
후보 용어       21
충돌 용어        5
금칙어 위반      1

현재 가장 중요한 용어:
黒架 → 흑가 / 쿠로카 충돌
```

이 영역은 NovelTrans의 정체성이다.
다른 번역 도구는 진행률을 보여주지만, NovelTrans는 **용어집의 건강 상태**를 보여준다.

## 9. Glossary Lab: 용어집 연구실

용어집 관리는 설정 메뉴 안에 숨어 있으면 안 된다.
별도의 주요 공간으로 분리한다.

```text
Glossary Lab

┌─ Queue ────────────────┬─ Term Detail ───────────────────────────────┐
│ > 黒架        충돌       │ 원어: 黒架                                  │
│   聖印        후보       │ 유형: 인명 후보                              │
│   魔導炉      후보       │ 등장: 12회 / 4화                              │
│   白の王      확정       │                                               │
│   影縫い      금칙 확인   │ 번역 후보:                                    │
│                         │  1. 흑가        7회                            │
│                         │  2. 쿠로카      5회                            │
│                         │                                               │
│                         │ 추천: 흑가                                     │
└─────────────────────────┴──────────────────────────────────────────────┘

[Enter] 추천 확정   [E] 직접 수정   [L] 잠금   [F] 금칙어 추가   [S] 넘김
```

용어집 UX는 “표 편집”이 아니라 “검토 큐”에 가깝게 만든다.

사용자는 후보를 하나씩 처리한다.

* 확정
* 수정
* 잠금
* 금칙어 추가
* 폐기
* 나중에 보기

## 10. 용어집 상태를 게임처럼 만들지 말고, 연구실처럼 만든다

용어집을 너무 게임화하면 제품이 가벼워 보인다.
대신 연구실 느낌을 준다.

표현 예시:

```text
용어집 건강도
일관성 92%
충돌 5개
검토 대기 21개
잠금 적용률 78%
```

또는:

```text
이번 번역에서 새로 발견된 용어

+ 聖印
+ 魔導炉
+ 影縫い
```

핵심은 사용자가 “용어집이 자라고 있다”는 느낌을 받는 것이다.

## 11. Review Desk: 검수 작업대

QA 결과는 긴 리포트가 아니라 처리 가능한 카드 큐로 보여준다.

```text
Review Desk

┌─ Issues ────────────────────┬─ Detail ───────────────────────────────┐
│ > 12화  일본어 잔존           │ 12화 / 84번째 문단                      │
│   18화  이름 흔들림           │ 감지된 원문: それでも、私は行く。          │
│   24화  길이 비율 이상         │ 번역문에 일본어 문장이 남아 있습니다.       │
│   31화  금칙어 위반            │                                             │
│                              │ [O] 번역문 열기                            │
│                              │ [M] 무시 처리                              │
│                              │ [T] 해당 화 재번역                         │
└──────────────────────────────┴─────────────────────────────────────────┘

[Enter] 열기   [T] 재번역   [M] 무시   [G] 관련 용어 보기
```

QA는 “보고서”가 아니라 “처리해야 할 inbox”다.

## 12. Export Room: 결과물 제작실

결과 파일 생성은 단순 명령이 아니라 결과물 제작 화면으로 만든다.

```text
Export Room

출력 형식
[●] TXT
[●] EPUB

EPUB 옵션
[●] 목차 생성
[●] 용어집 부록
[●] 작가 후기 포함
[○] 세로쓰기
[○] 표지 이미지

미리보기
제목: 마도서관의 사서
화수: 100화
용어집 부록: 128개 항목
예상 파일명: madoseogwan.epub

[Enter] 생성   [P] 미리보기   [O] 출력 폴더 열기
```

Export Room의 목표는 “파일이 생겼다”가 아니라 “책이 완성됐다”는 느낌을 주는 것이다.

## 13. Command Palette

깊은 메뉴 탐색을 줄이기 위해 command palette를 제공한다.

키:

```text
Ctrl+K 또는 :
```

예시:

```text
Command Palette

› glossary

- 용어집 열기
- 충돌 용어만 보기
- 후보 용어 검토
- 용어집 JSON 내보내기
- 현재 화의 관련 용어 보기
```

고급 사용자는 메뉴를 타지 않고 바로 기능을 실행할 수 있다.


## 15. 사용자 경험 원칙

### 15.1 메뉴보다 상태를 보여준다

사용자는 기능 목록보다 현재 프로젝트 상태를 먼저 봐야 한다.

### 15.2 설정은 숨기되, 통제권은 준다

기본값은 추천하고, 고급 사용자는 언제든 수정할 수 있게 한다.

### 15.3 용어집은 항상 가까이에 둔다

용어집은 별도 부가 메뉴가 아니다. Studio, Review, Export 어디서든 접근 가능해야 한다.

### 15.4 실패를 정상 흐름으로 취급한다

실패 화면은 에러 메시지가 아니라 복구 화면이어야 한다.

```text
3개 화 번역 실패

원인:
- 24화: API timeout
- 31화: 응답 형식 오류
- 45화: 빈 번역 결과

추천:
[R] 실패한 화만 재시도
[S] 실패 화 건너뛰고 export
[L] 로그 보기
```

## 16. 최종 UX 문장

NovelTrans는 사용자가 설정 메뉴를 따라가는 번역 CLI가 아니다.
하나의 작품을 책장에 꽂고, 작업실에서 번역하고, 연구실에서 용어를 다듬고, 검수 작업대에서 문제를 처리하고, 마지막으로 Export Room에서 읽을 수 있는 책으로 완성하는 터미널 번역 스튜디오다.

제품의 중심은 “번역 실행 버튼”이 아니라 “작품 프로젝트”와 “성장하는 용어집”이다.
